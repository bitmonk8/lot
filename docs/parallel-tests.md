# Spec: Parallel Test Execution

## Problem

All CI test jobs run with `--test-threads=1` (see `DESIGN.md` line 294,
`.github/workflows/ci.yml` lines 98, 116, 136). The stated reason is
"integration tests modify system state." This serialization is the
primary CI bottleneck: Test (Windows) takes ~1m54s wall clock, with 47s
in `cargo test` alone. Parallel execution would reduce this
proportionally to the longest individual test.

## Analysis: Why Serialization Is Not Necessary

Each shared resource is examined below. None require global test
serialization.

### AppContainer Profile Names (Windows)

Profile names are generated by `unique_profile_name()`
(`appcontainer.rs:70-84`) using four components:
`lot-{pid}-{tick}-{qpc}-{seq}`, where `seq` is an `AtomicU64` counter.
Within a single test process (same PID), the atomic counter guarantees
uniqueness across threads. No two tests will collide on profile names.

### Sentinel Files (Windows)

Sentinel files are written to `std::env::temp_dir()` with filenames
derived from the profile name: `lot-sentinel-{profile_name}.txt`
(`sentinel.rs:23-25`). Since profile names are unique per spawn,
sentinel file paths are unique per spawn. Two concurrent tests will
never write to the same sentinel file.

### NUL Device ACE (Windows)

The NUL device (`\\.\NUL`) has a system-wide DACL modified by
`grant_nul_device()` (`nul_device.rs`). This is NOT an issue because:

1. The grant is performed once during `lot setup` (CI step
   "Setup AppContainer prerequisites"), before any tests run.
2. The grant is idempotent and persistent across reboots.
3. No test modifies or revokes the NUL device ACE.
4. Tests only read-check the ACE via `nul_device_accessible()`.

Concurrent tests all observe the same pre-existing ACE. No race.

### Traverse ACEs (Windows)

`grant_traverse()` (`traverse_acl.rs`) adds traverse ACEs to ancestor
directories of policy paths. For tests:

1. Each test creates its own `TempDir` under `test_tmp/`
   (`common/mod.rs:12-19`), producing unique paths.
2. Ancestor directories of those unique paths are: `test_tmp/`,
   the workspace root, `C:\UnitySrc`, and `C:\`. These are shared.
3. However, `grant_traverse()` is idempotent — it checks for an
   existing ACE before writing (`traverse_acl.rs:169-171`). Two tests
   granting the same ancestor simultaneously both succeed (one writes,
   the other no-ops).
4. Traverse ACEs are never revoked by tests. They accumulate harmlessly.

No race.

### Policy Path ACLs (Windows)

`spawn()` grants read/write ACEs on policy paths (test-specific
`TempDir` contents) and revokes them on cleanup. Since each test
operates on its own `TempDir`, no two tests modify ACLs on the same
paths.

### `cleanup_stale()` (Windows)

`cleanup_stale()` scans `std::env::temp_dir()` for sentinel files whose
owning PID is dead, then restores their ACLs and deletes the sentinel.
This is the one genuine concern:

- A concurrent test could be mid-write of a sentinel file while
  `cleanup_stale()` scans the same directory.
- `cleanup_stale()` filters by PID liveness — it skips sentinels owned
  by the current process. Since all tests run in the same process,
  `cleanup_stale()` will skip all active sentinels.

The existing `TEST_LOCK` mutex in `appcontainer.rs:990-992` serializes
unit tests that call `cleanup_stale()`. Integration tests that call
`lot::cleanup_stale()` (`integration.rs:456, 1148`) would need the same
protection — but only against each other, not against all tests.

**Fix:** Either (a) wrap the two integration tests that call
`cleanup_stale()` in a shared mutex, or (b) accept the benign race
since `cleanup_stale()` already skips same-PID sentinels.

### Cgroups (Linux)

`CgroupGuard::new()` (`cgroup.rs:98+`) creates sibling cgroups under
the parent of the current cgroup with names like
`lot-sandbox-{pid}-{nanos}` plus a retry suffix on collision. Each
sandbox gets a unique cgroup path. Cleanup (kill + rmdir) is scoped to
that path. No two tests share a cgroup directory.

The parent cgroup's `subtree_control` file is read-only during tests
(set up by CI before tests run). No contention.

### Namespaces and seccomp (Linux)

`fork()` + `unshare()` creates per-child namespaces. These are
inherently isolated — the kernel guarantees no cross-namespace
interference. `seccomp` filters are per-thread/process. No shared state.

### Seatbelt (macOS)

`sandbox_init()` applies a per-process seatbelt profile in the forked
child. The parent (test process) is never sandboxed. Each child gets its
own profile string. No shared state.

### Resource Limits (macOS)

`setrlimit()` is applied in the forked child, not the parent. No effect
on other tests.

## Implementation Plan

### Step 1: Remove `--test-threads=1` from CI

In `.github/workflows/ci.yml`, remove `--test-threads=1` from all three
test jobs (lines 98, 116, 136). Replace with default parallelism (Rust
test harness uses number of CPUs).

### Step 2: Guard `cleanup_stale()` calls in integration tests

The two integration tests that call `lot::cleanup_stale()`
(`integration.rs:456, 1148`) should use `#[serial]` from the
`serial_test` crate, or a shared `std::sync::Mutex` similar to the
existing `TEST_LOCK` in `appcontainer.rs:990-992`. This serializes only
these two tests against each other, not the entire suite.

### Step 3: Update DESIGN.md

Remove or revise the statement at line 294:
> Tests run with `--test-threads=1` (integration tests modify system state).

Replace with a note that tests are parallel-safe by design: unique
profile names, unique temp dirs, idempotent ACE grants.

### Step 4: Verify

Run `cargo test -p lot --features tokio -- --nocapture` (without
`--test-threads=1`) on all three platforms and confirm no flaky failures
across multiple runs.

## Risks

1. **Undiscovered shared state.** If a test modifies global state not
   identified above, parallel execution could cause flaky failures. The
   fix is to identify and isolate that specific test, not to re-serialize
   everything.

2. **CI resource contention.** Parallel tests may compete for CPU/memory
   on CI runners, potentially causing timeout tests to be less reliable.
   Timeout thresholds may need adjustment.

3. **Cgroup contention on Linux.** Multiple concurrent `CgroupGuard`
   instances writing to sibling cgroups under the same parent should be
   safe, but filesystem operations on cgroupfs are kernel-serialized.
   Unlikely to cause failures but worth monitoring.
